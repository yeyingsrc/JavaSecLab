其他漏洞 - 跨站请求伪造 

  CSRF(跨站请求伪造)：攻击者诱导已登录用户访问恶意页面或触发恶意请求，浏览器会自动携带目标站点 Cookie 等凭证，导致服务端误以为请求来自用户本人，从而执行转账、改密、绑定账号等敏感操作。
  触发条件：目标操作依赖 Cookie/Session 自动认证，且缺少不可预测的 CSRF Token、Origin/Referer 校验、SameSite Cookie 或二次确认等防护。GET 请求执行状态变更会显著放大风险。

安全场景：CSRF Token防护

  • 转账操作
tips
安全编码建议：
  1、真实业务中敏感操作使用 POST/PUT/DELETE 等非 GET 方法，并开启框架内置 CSRF 防护。
  2、每个敏感请求携带不可预测的 CSRF Token，服务端与 Session 中的 Token 做校验。
  3、Token 不应出现在可被第三方直接构造的固定 URL 中，真实业务中通常放在表单隐藏字段或自定义请求头。
  4、配合 SameSite Cookie、二次确认、短有效期和操作审计提升防护强度。

tips
辅助检测说明：
  Origin/Referer 可以作为 CSRF Token 之外的补充校验，但不应作为唯一防线。
  校验时应解析 URL 后精确比较协议、域名和端口，不要使用 startsWith 这类字符串前缀判断。
  某些隐私策略、跳转链路或客户端环境可能缺失 Referer，因此缺失时应拒绝敏感操作或走二次确认。
