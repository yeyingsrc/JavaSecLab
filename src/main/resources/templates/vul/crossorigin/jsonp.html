其他漏洞 - 跨源安全问题 

  同源策略(Same Origin Policy)：浏览器限制脚本读取不同源的响应内容；协议、域名、端口三者完全一致才属于同源。script 标签加载脚本不受同源读取限制，这是 JSONP 能工作的基础。
  JSONP(JSON with Padding)：服务端把 JSON 数据包进 callback(...) 形式，让页面通过 script 标签跨源加载执行。它只支持 GET，缺少现代 CORS 的细粒度控制，不适合承载账号、权限、订单等敏感数据。

漏洞场景：JSONP劫持

接口把敏感数据包进未校验的 callback 参数，任意站点都可以通过 script 标签读取

tips
问题点：
  1、JSONP 响应会被浏览器当作脚本执行，不受普通 XHR/fetch 的同源读取限制。
  2、callback 未校验可能造成脚本注入；即使校验了 callback，敏感数据仍可能被任意站点通过 script 标签读取。
  3、JSONP 只适合公开只读数据，不能用于登录态相关或含隐私信息的接口。

安全场景：callback校验与公开数据

校验 callback 格式，并且只返回公开数据；敏感接口应迁移到受控 CORS

tips
安全编码建议：
  1、能不用 JSONP 就不用 JSONP，优先迁移到 CORS、鉴权和标准 JSON API。
  2、必须保留 JSONP 时，callback 只允许合法 JavaScript 标识符或受控白名单函数名。
  3、JSONP 接口只返回公开、只读、可缓存数据，不返回用户身份、权限、订单、Token 等敏感信息。
  4、设置 application/javascript;charset=UTF-8 和 X-Content-Type-Options: nosniff，降低错误解析风险。
